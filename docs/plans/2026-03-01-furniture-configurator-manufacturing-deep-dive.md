# Furniture Configurator Manufacturing Deep Dive

**Date:** 2026-03-01
**Status:** Research and planning
**Scope:** Furniture Configurator -> cutlist -> drawings -> order/job card -> factory execution

## Why This Matters

The current Furniture Configurator already proves the value of parametric furniture definition inside Unity ERP, but it stops too early in the manufacturing chain. It can generate panels and hand them to the cutlist flow, yet it does not persist a true manufacturing package that can follow the configured unit into orders, drawings, job cards, and staff execution.

The next phase should not be treated as a UI enhancement. It is a manufacturing handoff problem.

## Verified Current State

### Product authoring

- The configurator route exists at [`app/products/[productId]/configurator/page.tsx`](../../app/products/[productId]/configurator/page.tsx).
- The current implementation is cupboard-only and uses a template pattern documented in [`docs/features/furniture-configurator.md`](../features/furniture-configurator.md).
- The live preview is generated directly from config in [`components/features/configurator/CupboardPreview.tsx`](../../components/features/configurator/CupboardPreview.tsx).
- The part list is generated by [`lib/configurator/templates/cupboard.ts`](../../lib/configurator/templates/cupboard.ts).

### Persistence

- Saving from the configurator writes only cutlist groups through [`app/api/products/[productId]/cutlist-groups/route.ts`](../../app/api/products/[productId]/cutlist-groups/route.ts).
- Those groups are stored in `public.product_cutlist_groups` with org-scoped access already in place.
- The saved payload is a flat JSONB parts array; it does not store the source template, full parameter set, drawing artifact, or manufacturing instructions.

### Cutlist builder handoff

- “Save & Open Cutlist Builder” pushes the generated parts into the product cutlist builder route.
- The builder loads product cutlist groups, flattens them, and edits them as generic cutlist parts via [`app/products/[productId]/cutlist-builder/page.tsx`](../../app/products/[productId]/cutlist-builder/page.tsx) and [`lib/configurator/cutlistGroupConversion.ts`](../../lib/configurator/cutlistGroupConversion.ts).
- Per-row edge editing exists through [`components/features/cutlist/primitives/EdgeBandingPopover.tsx`](../../components/features/cutlist/primitives/EdgeBandingPopover.tsx).
- Current edge controls support per-part edits and “Clear All” inside a single part popover, but there is no explicit bulk “reset banding for all configurator parts” flow.

### Manufacturing and job cards

- Job cards currently model labor execution, not a manufacturing packet. The core tables are `job_cards` and `job_card_items`.
- The order job-card flow is centered on BOL/job issuance, not on configured drawings or cut parts.
- The job-card PDF already supports an optional drawing image through [`components/features/job-cards/JobCardPDFDocument.tsx`](../../components/features/job-cards/JobCardPDFDocument.tsx), but the current job-card page does not supply one.

### Reusable image/drawing infrastructure

- Quotes already have non-destructive image crop/annotation support on `quote_attachments` (`crop_params`, `annotations`, `display_size`).
- The existing annotation/editor work is a strong candidate for reuse when drawing markups become necessary.

## Critical Gaps

### 1. The configurator saves parts, not a configuration

Today the system loses the most important information after save:

- template identity
- canonical parameter values
- derived construction rules
- preview geometry
- drawing artifacts
- intent metadata such as part role

That means downstream consumers only receive anonymous panel rows.

### 2. The saved parts are too lossy for robust drawing generation

`CutlistPart` carries dimensions and banding, but not enough semantic context to generate reliable drawings or downstream manufacturing instructions.

Missing examples:

- `part_role` such as `top`, `base`, `side_left`, `shelf`, `door_left`
- assembly grouping
- view geometry or callout anchors
- finish-face metadata
- default machining notes
- edge-treatment reason or preset origin

### 3. Thickness handling is currently inconsistent

This is the most important technical mismatch found in the current code.

- The configurator allows `materialThickness` values `16`, `18`, and `25`.
- The generator correctly derives laminated thickness as `T * 2`.
- But the configurator save path always maps parts into cutlist group board types `16mm` and `32mm-both`.

That means:

- 18mm cupboards still save as `16mm` / `32mm-both`
- 25mm cupboards still save as `16mm` / `32mm-both`
- edge-banding expectations become wrong
- builder semantics drift from the actual product definition

This is not just a polish issue. It can produce wrong edging, wrong costing, and wrong factory interpretation.

### 4. No manufacturing artifact reaches the job card

The system can already print a job card with a drawing, but there is no persisted artifact chain from:

configurator -> order line -> job card -> staff member

### 5. Edge-banding correction is too manual for configurator-driven work

Current capability:

- per-row edits
- per-row “Clear All”
- apply edging material to all rows

Missing capability:

- reset banding for all configurator parts
- reset banding for selected rows
- reapply configurator defaults after edits
- bulk change by part role
- visual diff between template defaults and manual overrides

### 6. No immutable order-line snapshot exists for configured furniture

The orders documentation already identifies a broader configurable-products gap around `order_details.selected_options`. For the configurator, the gap is larger: a configured order line must carry a frozen manufacturing snapshot, otherwise changes on the product later can silently alter what the factory sees.

## Recommended Product Model

The configurator should become a two-layer system.

### Layer 1: Product master definition

Purpose: define the standard configurable furniture product.

Recommended persisted shape:

- `product_id`
- `org_id`
- `template_key`
- `config_version`
- `config_json`
- `parts_json`
- `drawing_manifest_json`
- `preview_svg`
- `manufacturing_notes`
- `is_active`

Recommended new table name:

- `product_configurator_versions`

Why versioned:

- product definitions will evolve
- old orders must stay reproducible
- drawings and part defaults must be traceable to the version issued

### Layer 2: Order-line manufacturing snapshot

Purpose: freeze exactly what was sold and must be built.

Recommended persisted shape:

- `order_detail_id`
- `org_id`
- `product_id`
- `source_configurator_version_id`
- `template_key`
- `config_json`
- `parts_json`
- `drawing_manifest_json`
- `override_notes`
- `status`

Recommended approach:

- store a dedicated snapshot when a configured product is added to a quote or order
- never regenerate from the latest product definition at factory execution time

This should be generic enough to support other future configurable products, not only cupboards.

## Recommended Drawing Strategy

Use both config-driven generation and stored artifacts.

### Source of truth

- The canonical source should remain `template_key + config_json`.
- Drawings should be regenerated from config by deterministic code.

### Persisted artifacts

Also persist generated artifacts for stable downstream use:

- front elevation SVG
- side elevation SVG
- optional exploded panel view SVG
- print-ready PNG or PDF derivatives
- metadata for dimensions, labels, and revision

### Why both are needed

Config alone is best for deterministic regeneration.
Artifacts are best for:

- PDFs
- attachments
- offline review
- job-card printing
- future annotation workflows

### Reuse opportunity

When annotations are needed, reuse the quote image editor model:

- crop data
- arrow/callout annotations
- display sizing

That avoids inventing a second drawing-markup format.

## Recommended Cutlist and Edge-Banding Strategy

### 1. Enrich generated parts with semantic metadata

Extend generated part payloads with at least:

- `part_role`
- `assembly_key`
- `source_template`
- `source_version`
- `default_band_edges`
- `banding_override_state`
- `finish_face`
- `material_thickness_mm`
- `final_thickness_mm`

Without this, downstream bulk editing and drawings remain fragile.

### 2. Replace hardcoded board types with thickness-aware modeling

The current `16mm` / `32mm-both` / `32mm-backer` model is too rigid for the configurator.

Recommended direction:

- preserve the existing board types for legacy/manual builder flows
- introduce thickness-aware lamination metadata for configurator-originated parts
- calculate final thickness from actual board thickness and lamination count

Example:

- 18mm single -> 18mm edge family
- 18mm same-board pair -> 36mm edge family
- 25mm same-board pair -> 50mm edge family

This aligns with the existing unfinished work already captured in [`docs/plans/cutlist-improvements.md`](cutlist-improvements.md).

### 3. Add bulk banding controls

Minimum viable additions:

- `Reset banding for all`
- `Reset banding for selected`
- `Apply configurator defaults`
- `Apply current row banding to selected`
- `Clear all banding overrides`

Preferred grouping axes:

- all rows
- selected rows
- rows sharing `part_role`
- rows sharing `assembly_key`

### 4. Track override provenance

Every part should know whether its current edge state is:

- generated default
- user-edited in configurator
- user-edited in cutlist builder
- imported from snapshot

That makes later resets and audits predictable.

## Recommended Manufacturing Flow

### Target flow

1. Product author defines a configurator-backed product master.
2. User configures the item and saves a product version.
3. Sales or operations creates a quote/order line from that product.
4. The selected configuration is frozen into an order-line manufacturing snapshot.
5. The snapshot generates or references:
   - panel parts
   - edge-banding schedule
   - drawing artifacts
   - manufacturing notes
6. Labor planning / job issuance creates job cards from that snapshot.
7. The staff-facing job card includes:
   - drawing
   - part summary
   - critical dimensions
   - edge-banding notes
   - revision/version reference

### Important principle

The job card should not derive manufacturing truth from the mutable product record at print time. It should use the frozen order-line snapshot.

## Job Card Integration Recommendation

### Short-term path

Use the existing job-card PDF capability and attach a single primary drawing artifact from the configuration snapshot.

Short-term implementation:

- resolve `drawingUrl` from the order-line manufacturing snapshot
- pass it into the existing `JobCardPDFDownload`

### Medium-term path

Introduce a richer manufacturing packet attached to job cards.

Recommended new table:

- `job_card_attachments`

Recommended fields:

- `job_card_id`
- `org_id`
- `source_type`
- `source_id`
- `attachment_type`
- `file_url`
- `mime_type`
- `metadata`

Useful attachment types:

- `configurator_drawing`
- `configurator_packet`
- `cutting_diagram`
- `assembly_reference`

### Why not put everything directly on `job_cards`

Because one job card can end up with multiple relevant artifacts, and a single `drawing_url` field will not scale once:

- a card covers multiple products
- reprints create revisions
- cutting diagrams are separated from assembly drawings

## Fit With the Rest of the App

### Orders

The configurator should slot into the same configurable-products story already identified in [`docs/domains/orders/orders-master.md`](../domains/orders/orders-master.md), but it needs a stronger snapshot model than plain `selected_options`.

### Quotes

Configured furniture should eventually support:

- quote-line configuration
- cutlist preview
- optional drawing attachment to the quote item

The quote attachment editor is likely reusable once drawing markups matter.

### Components / BOM / BOL

The configurator does not replace BOM/BOL. It should feed them.

Recommended stance:

- configurator defines the manufactured panel package
- BOM defines material/component dependencies beyond panels
- BOL defines labor steps
- job cards issue work from the frozen order-line snapshot plus BOL context

### Factory floor

Factory-floor and job-action work should consume configurator outputs only through job cards or manufacturing packets, not through ad hoc product lookups.

## Tenancy and Access

Any new configurator persistence must follow the existing organization model.

Rules for new tables:

- include `org_id`
- enforce org membership under RLS
- verify parent-child org consistency
- check module entitlement at relevant server routes

Good current examples:

- `product_cutlist_groups`
- `organization_module_entitlements`
- module access guard usage in [`app/api/products/[productId]/cutlist-groups/route.ts`](../../app/api/products/[productId]/cutlist-groups/route.ts)

## Recommended Delivery Phases

### Phase 1: Stabilize the current configurator output

- Add semantic part metadata.
- Fix the thickness-model mismatch between configurator and cutlist builder.
- Add bulk edge-banding reset/apply flows.
- Persist the actual configurator definition, not only cutlist groups.

### Phase 2: Generate manufacturing artifacts

- Generate deterministic SVG drawing views from config.
- Persist artifact manifest plus exportable PNG/PDF variants.
- Add versioning and revision metadata.

### Phase 3: Freeze configured order-line snapshots

- Add a generic configuration snapshot to quote/order lines.
- Ensure the order uses a frozen copy of config, parts, and drawings.
- Make the snapshot the source for production handoff.

### Phase 4: Attach manufacturing packets to job cards

- Resolve drawing artifacts into job-card PDFs.
- Add job-card attachment support if needed.
- Surface revision/version context to planners and factory staff.

### Phase 5: Expand templates and manufacturing intelligence

- desk
- pedestal
- bookshelf
- richer machining notes
- assembly sequences
- part labels / QR-ready identifiers

## Suggested Implementation Order

If development starts tomorrow, the highest-value order is:

1. Fix thickness-aware downstream representation.
2. Persist configurator versions and snapshots.
3. Add bulk edge-banding reset/apply.
4. Generate stored drawing artifacts.
5. Feed drawing artifacts into job-card PDFs.
6. Add order-line manufacturing snapshot and factory issuance flow.

## Immediate Build Recommendations

### Recommendation 1

Do not extend the current `product_cutlist_groups.parts` JSON into an all-purpose configurator store. Keep it for cutlist-builder compatibility, but add a dedicated configurator version table.

### Recommendation 2

Treat drawing generation as a first-class output of the configurator, not a later report.

### Recommendation 3

Add `part_role` now. Many future requirements depend on it:

- banding presets
- drawings
- assembly notes
- QC checklists
- selective rework

### Recommendation 4

Keep job cards document-driven. Staff should receive a printable packet with a stable revision, not a live-rendered view that can drift.

## Open Questions To Confirm Later

- Should one job card carry one configured unit, or can multiple configured units share a packet?
- Do we want one manufacturing snapshot per `quote_item`, per `order_detail`, or both?
- Should configurator-driven drawings live in a dedicated storage path or reuse product image/document patterns?
- Do we want operator annotations on drawings before job-card issue, or only read-only generated drawings in phase 1?
- Should banding presets be editable at template level, part-role level, or both?
- Is machining metadata needed in phase 1, or can it wait until after drawing + job-card integration is stable?

## Morning Starting Point

The cleanest first implementation slice is:

1. Introduce a dedicated configurator version/snapshot table with `org_id`.
2. Store `template_key`, `config_json`, enriched `parts_json`, and a generated preview SVG.
3. Add `part_role` and thickness-aware metadata to generated parts.
4. Add bulk banding reset/apply controls in the cutlist builder.
5. Wire one persisted drawing artifact into the existing job-card PDF.

That slice establishes the manufacturing chain without trying to solve the full quoting and order aggregation story in one pass.
